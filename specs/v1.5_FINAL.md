# BC Backcountry Scout — v1.5 Final Spec

**Status:** Production (May 2026)  
**Bot:** @bc_scout_bot on Telegram  
**Deploy:** Self-hosted Oracle Cloud (single instance, systemd service)

---

## Overview

A Telegram bot that pulls road conditions, weather, fires, wildlife, air quality, and park advisories for a BC backcountry destination in a single concise report.

**User flow:**
1. `/scout` + destination name
2. Bot confirms destination (with optional starting point override)
3. Generate comprehensive report (2-5s, parallel fetches)
4. User can request: 3-day forecast, avalanche forecast, offline report, or scout new destination

- **Safety Block:** Road closures, active fires, fire bans, wildlife advisories, park advisories, air quality

- **Driving Block:** Distance + ETA with traffic, webcam link (nearest within 30km)

### Known Limitations

**Fire Bans:**
- **Issue:** Oracle Cloud IPs are IP-reputation filtered by BC government geospatial services (openmaps.gov.bc.ca, BCWS ArcGIS)
- **Current Behavior:** Returns empty list, no user-facing error (graceful degradation)
- **Solutions:**
  1. (Preferred) Contact BC government OGC to allowlist Oracle Cloud IP/ASN ranges
  2. Proxy through allowlisted IP (requires external infrastructure)
  3. Migrate to non-cloud hosting (Linode, DigitalOcean, etc.)
  4. Accept graceful degradation (current state)
- **See:** `TROUBLESHOOTING.md` for detailed guidance

**Webcam Fallback:**
- DataBC WFS primary, Open511 fallback (404 known, rarely used)
- Handles gracefully: returns None if both fail

## Architecture

### Request Flow
```
/scout message
  ↓
geocode_destination(query_text)  → validate coordinates → confirm with user
  ↓
scout_confirm callback
  ↓
run_all_fetchers() [parallel, 8s timeout each]
  ├─ DriveBC (road events) [5min cache]
  ├─ Open-Meteo (weather, 3-day) [15min cache]
  ├─ BCWS ArcGIS (wildfires, fire bans) [5min cache, fallback WFS]
  ├─ BC Parks REST (advisories) [15min cache]
  ├─ Open-Meteo (AQHI) [5min cache]
  ├─ avalanche.ca XML (forecast) [seasonal]
  ├─ Google Maps (ETA) [no cache, per-request]
  ├─ DataBC WFS (webcams) [24h cache, fallback Open511]
  └─ Environment Canada RSS (alerts) [per-fetch]
  ↓
assemble_report() → HTML formatted
  ↓
Telegram message + keyboard (3-day, avalanche, offline, new scout)
```

### Data Sources
| Source | Endpoint | Fallback | Cache |
|--------|----------|----------|-------|
| Roads | DataBC WFS | None | 5min |
| Weather | Open-Meteo API | None | 15min |
| Wildfires | BCWS ArcGIS | DataBC WFS | 5min |
| Fire Bans | BCWS ArcGIS | DataBC WFS | [IP-blocked] |
| Avalanche | avalanche.ca XML | None | session |
| Webcams | DataBC WFS | Open511 | 24h |
| Park Advisories | BC Parks REST API | None | 15min |
| AQHI | Open-Meteo | None | 5min |
| Alerts | Environment Canada RSS | None | per-fetch |
| Geocoding | Google Maps + GNWS | GNWS only | None |
| ETA | Google Maps | None | None |

### Caching Strategy
- **Per-location caching:** Coordinates are cache key (tuple lat/lon)
- **TTLs vary:** 5–1440 minutes depending on data volatility
- **No global invalidation:** Data naturally refreshes on TTL expiry
- **Unbounded aqhi.py cache:** Medium-severity issue (grows with unique coordinates)

## Performance

**Typical scout operation:** 2–5 seconds
- Parallel fetches: ~300ms each (8s timeout per fetcher)
- Report assembly: <100ms
- Bottleneck: Google Maps ETA API (~1.5s) and EC alerts fetch (~1.5s)

**Memory:** ~50–80 MB steady state (excludes cache growth)

**Cache memory:** ~5–10 MB with typical BC-scale user base

## Known Issues

1. **Fire bans intermittently empty** — Oracle Cloud IP filtering (documented)
2. **AQHI cache unbounded** — Should use LRU eviction (low risk)
3. **Webcam fallback rarely works** — Open511 endpoint returns 404
4. **No rate limiting** — Relies on API limits; could DOS upstream services with malformed input
5. **Multi-process deployment unsupported** — Session file race conditions

**v1.5 is production-ready** with graceful degradation for known issues (fire bans). The codebase is clean, well-tested on utilities, and safe for single-process deployment on Oracle Cloud. Security fixes applied in final audit round.

Key learnings:
- Geospatial API fragility (IP filtering, schema inconsistency)
